# BBB Group 13: Production Release Archive

This group creates the final archive process for a production release. It is the record of what was shipped, what was verified, who approved it, and where the supporting evidence lives.

## Implemented

- Added production release archive checklist.
- Added archive folder structure and naming convention.
- Added package inclusion and exclusion rules.
- Added release manifest template.
- Added archive evidence checklist for deployment, QA, privacy, and support.
- Added release-retention and retrieval guidance.
- Added reusable archive template: `deploy/bbb-live-release-archive-template`.
- Added verification SQL: `src/moodle/local_prequran/sql/verify_group_13_release_archive.sql`.

## Archive Goal

The release archive must answer these questions later:

1. Which Moodle plugin files were released?
2. Which verification scripts were included?
3. Which documents were included?
4. Which production checks passed?
5. Which known issues were accepted?
6. Who approved release, rollback readiness, and child-safety readiness?
7. Where are screenshots, SQL results, and sign-off notes stored?

## Archive Naming

Use this naming format:

```text
quraan-academy-bbb-live-release-YYYYMMDD-vNN
```

Example:

```text
quraan-academy-bbb-live-release-20260513-v01
```

Use one archive per production release. Do not overwrite a previous archive.

## Include In Archive

Include:

- `src/moodle/local_prequran`
- `src/moodle/local_hubredirect`
- Group 1 through Group 13 documentation.
- All verification SQL scripts.
- Sanitized screenshots from diagnostics, smoke tests, and dashboards.
- Production deployment checklist result.
- Final consistency audit result.
- Release sign-off result.
- Rollback decision owner and process.

## Exclude From Archive

Do not include:

- BBB shared secret.
- Moodle `config.php`.
- Raw database dumps unless separately encrypted and approved.
- Raw BBB recordings.
- Session tokens or join URLs.
- Child private notes.
- Parent messages or replies with personally identifiable details.
- Browser cookies.
- Server logs containing secrets.
- Temporary files from phpMyAdmin exports.

## Retention

Keep release archives according to the academy retention policy. Recommended:

- Keep final release archives for at least 12 months.
- Keep pilot-only archives for at least 6 months.
- Keep incident-related release archives until the incident review is closed plus the required retention period.

Do not purge archives that are under active investigation, support review, privacy review, or legal hold.

## Retrieval Test

Before launch, one person who did not create the archive should confirm they can:

1. Find the archive.
2. Open the manifest.
3. Identify the deployed plugin folders.
4. Find the rollback plan.
5. Find the Group 12 consistency result.
6. Find the Group 13 archive result.
7. Confirm secrets and child private data are not included.
