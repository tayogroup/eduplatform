# Course Transcript Requirements

Purpose: define the requirements for an official course transcript that can be implemented after the course enrollment, EduForTomorrow, and institution workspace routing flows are stable.

The transcript is not just a course history table. It is a controlled academic record that summarizes a learner's enrollment, completion, performance, attendance, teacher evaluation, and credential status across one consumer/workspace context. It must be accurate enough for parents, teachers, administrators, partner institutions, and future scholarship or placement reviews.

## Goals

- Give each student and linked parent a trusted view of completed, active, dropped, transferred, and pending courses.
- Give workspace admins a printable/exportable official transcript for one student.
- Support EduForTomorrow, Quran Academy, and institution-branded transcripts without leaking the wrong brand, domain, workspace, or teacher data.
- Reconcile platform course offerings with Moodle enrollment, Moodle grades, lesson progress, live-session attendance, teacher evaluations, certificates, and admin adjustments.
- Preserve a defensible audit trail for transcript-affecting changes.
- Keep the first implementation practical: read from existing course offering and Moodle records where possible, then add transcript-specific tables only for snapshots, overrides, publication, holds, and exports.

## Scope

In scope:

- Student transcript page for the authenticated student.
- Parent transcript page for linked children.
- Workspace admin transcript viewer and export flow.
- Official PDF export and CSV/admin export.
- Transcript snapshots for issued official records.
- Consumer/workspace branding, signatures, issue metadata, and verification code.
- Course-level transcript lines for approved/enrolled/completed/dropped/withdrawn offerings.
- Transcript holds, corrections, reissue workflow, and audit logging.

Out of scope for the first implementation:

- Automated transfer-credit evaluation from outside schools.
- Government-accredited transcript claims unless the institution has separate legal approval.
- Fully self-service third-party delivery.
- Complex GPA weighting unless a workspace explicitly enables graded programs.
- Cross-consumer combined transcripts unless a platform admin intentionally generates one.

## Transcript Types

The system should support these transcript views:

- Unofficial transcript: visible to student, linked parent, teacher with permitted student access, and workspace admins. It can include in-progress work and may update as data changes.
- Official transcript: generated by a workspace admin or authorized registrar role. It is a timestamped snapshot with a document ID, verification token, issuer, and locked data.
- Internal progress transcript: admin/teacher view that can include operational notes, pending sync problems, and incomplete evidence not shown on official exports.
- Placement transcript: shorter parent/student-facing record for advising, placement, and teacher handoff.

## Access And Permissions

- Students can view their own unofficial transcript only inside their current allowed consumer/workspace context.
- Parents can view transcripts only for children linked through existing guardian/consent relationships.
- Teachers can view transcript information only for students they teach or are assigned to inside the workspace.
- Workspace admins can view, correct, hold, issue, and export transcripts for students in their workspace.
- Consumer admins can view transcripts for workspaces under their consumer.
- Platform admins can support all consumers, but global views must label consumer and workspace clearly.
- Public verification pages must reveal only the minimum needed to confirm an official transcript: document ID, student name or masked identifier, issuer, issue date, status, and checksum/validity result.
- Direct URL access by guessed student ID, course ID, offering ID, document ID, or export path must fail outside the user's permitted scope.

## Transcript Header

Every transcript should show:

- Consumer or institution name.
- Workspace/campus/program name when applicable.
- Brand logo or seal when configured.
- Official support or registrar email.
- Student full name.
- Student account number or idnumber.
- Moodle user ID only in internal/admin views.
- Date of birth only if the workspace explicitly enables it and the viewer is authorized.
- Parent/guardian names only on internal or placement versions, not on default official exports.
- Transcript type: unofficial, official, internal, or placement.
- Issue date and generated-by user for official exports.
- Document ID and verification URL for official exports.
- Timezone used for dates.

## Course Line Requirements

Each transcript course line should include:

- Academic period or term label.
- Course title from the course offering.
- Course code/key.
- Moodle course shortname or external course code when available.
- Offering/workspace name when useful.
- Start date.
- End date.
- Enrollment status.
- Completion status.
- Final grade or outcome.
- Credits, hours, or level units if configured.
- Attendance summary when the course includes live sessions.
- Teacher of record or primary instructor when available.
- Certificate or credential earned.
- Notes visible on official export only when approved for transcript display.

Recommended normalized statuses:

- `requested`: enrollment request submitted but not approved.
- `approved_pending_sync`: approved locally but Moodle enrollment is not complete.
- `enrolled`: active Moodle enrollment exists.
- `in_progress`: active enrollment with learning activity started.
- `completed`: completion criteria met.
- `passed`: completion criteria met with passing outcome.
- `not_passed`: completion occurred but minimum passing criteria were not met.
- `withdrawn`: learner/admin ended participation before completion.
- `dropped`: enrollment was removed after approval.
- `cancelled`: pending request was cancelled before approval.
- `rejected`: request was rejected and should appear only in internal history by default.
- `transferred`: learner moved to another offering/course section.

Official transcripts should normally include enrolled, completed, passed, not passed, withdrawn, dropped, and transferred records. Requested, cancelled, rejected, and sync-problem records should default to internal/unofficial views unless a workspace intentionally includes them.

## Data Sources

The transcript resolver should combine these sources:

- `local_prequran_course_offering`: course title, key, Moodle course ID, dates, capacity context, visibility, and workspace.
- `local_prequran_course_enrol_req`: request, approval, enrollment sync, drop, and notes lifecycle.
- Moodle `course`, `enrol`, and `user_enrolments`: active enrollment proof.
- Moodle gradebook tables: grade items, final grades, grade scale, pass/fail threshold, and hidden/locked grade policy.
- Moodle course completion tables where enabled.
- Local lesson progress tables: unit/step completion, percent complete, completion time, quiz results, and practice evidence.
- Live-session tables: attendance, session completion, teacher notes, homework, follow-up, and parent-visible summaries.
- Certificate/credential tables when added.
- Transcript-specific snapshot/override tables when a record is issued or corrected.

The resolver should prefer official Moodle grade and completion data where available, but it should gracefully show "Not recorded" rather than inventing grades when a course has no gradebook setup.

## Completion And Grade Policy

Each workspace should define a transcript policy before issuing official records:

- Completion source: Moodle course completion, local unit completion, teacher/admin completion, or combined rule.
- Passing rule: grade threshold, rubric outcome, teacher outcome, attendance threshold, or no pass/fail.
- Grade display: letter grade, percentage, points, competency level, pass/fail, complete/incomplete, or hidden.
- Grade precision and rounding.
- Whether in-progress grades are shown on unofficial transcripts.
- Whether attendance affects completion or appears as a separate field.
- Whether late drops appear as withdrawn instead of dropped.
- Whether teacher notes can appear on official exports.

Default recommendation:

- Show official grades only after the course is marked completed, passed, not passed, withdrawn, or administratively finalized.
- Show in-progress percent and activity completion only on unofficial/internal views.
- Require an admin/registrar finalization step before official transcript issue if gradebook data is incomplete or conflicting.

## Official Transcript Snapshot

Official exports must be snapshot-based. Once issued, the record should not silently change when Moodle grades, names, course titles, branding, or historical enrollments change later.

Snapshot requirements:

- Store transcript header data at time of issue.
- Store each course line as issued.
- Store summary totals as issued.
- Store issuer user ID, consumer ID, workspace ID, student ID, issue timestamp, and reason.
- Store document ID, verification token hash, file hash, and status.
- Store replacement relationship if the transcript is corrected and reissued.
- Preserve revoked transcripts for audit, with a public verification status of revoked.

Suggested statuses:

- `draft`
- `issued`
- `revoked`
- `reissued`
- `expired` if the institution uses expiring verification links.

## Holds And Release Controls

The system should support transcript holds:

- Administrative hold: unresolved identity, consent, or account issue.
- Finance hold: optional, only if the institution enables billing integration later.
- Academic review hold: grade dispute, teacher review pending, suspected duplicate record.
- Data-quality hold: missing Moodle course mapping, missing grade policy, or inconsistent completion.

Finance hold behavior should follow the tuition and invoicing policy in `docs/student-tuition-payments-invoicing-requirements.md`, especially the rule that holds can block official transcript issue only when the workspace enables that policy.

When a hold exists:

- Students and parents may still see an unofficial transcript unless the hold says otherwise.
- Official export should be blocked or clearly marked unavailable.
- Admins should see the hold reason, created by, created at, and resolution note.
- Every hold add/remove action must be audited.

## Corrections And Appeals

Transcript-affecting changes should follow a correction workflow:

1. Admin opens a correction request for a specific student and course line or header field.
2. Admin records the reason and supporting note.
3. System stores the before/after values.
4. Authorized reviewer approves or rejects when the workspace requires dual control.
5. If an official transcript was already issued, the system marks the previous document as reissued or revoked and creates a new official snapshot.

Corrections should never edit an issued snapshot in place.

## Branding And EduForTomorrow Routing

Transcript pages and exports must use the active consumer/domain/workspace context:

- EduForTomorrow records show EduForTomorrow branding and support contact.
- Quran Academy records show Quran Academy branding and support contact.
- Institution records show the institution name, logo, workspace, support contact, and custom-domain verification URL when active.
- Official transcript verification URLs should use the issuing consumer or institution app/public domain, not a hard-coded quraantest URL.
- Email/download links must preserve the source consumer/domain.
- If a user belongs to multiple workspaces, the transcript view must show a workspace selector or clearly identify the active workspace.

No transcript page should combine courses from different consumers by default. A combined cross-consumer transcript should be a separate platform-admin action with clear warnings and labels.

## Privacy And Compliance

- Treat transcripts as education records with stricter access than ordinary progress dashboards.
- Minimize personally identifiable information on exports.
- Do not include private teacher notes, internal admin notes, support tickets, guardian contact details, recordings, or raw quiz attempts on official transcripts.
- Allow workspace policy to include parent-visible teacher comments only after explicit approval.
- Log every view, export, issue, revoke, reissue, and public verification check.
- Use signed or tokenized download links with expiration.
- Store generated PDF files outside public web roots or protect them through permission-checked file serving.
- Redact sensitive data in support/debug logs.

## Admin Workflow

Workspace admin transcript workflow:

1. Open student profile or Student Course History.
2. Select Transcript.
3. Review unresolved data-quality warnings.
4. Choose transcript type: unofficial preview, official issue, placement, or internal.
5. Confirm workspace transcript policy.
6. Resolve holds or missing required fields.
7. Preview generated transcript.
8. Issue official transcript when required.
9. Download PDF or CSV.
10. Optionally email a secure link to the student/parent.

The page should show warnings for:

- Course approved locally but not synced to Moodle enrollment.
- Moodle enrollment exists but no course offering record exists.
- Course offering has no grade policy.
- Gradebook has hidden or overridden grades.
- Course completion is missing but local progress says complete.
- Student appears in multiple active workspaces.
- Existing official transcript has been issued after the latest course change.

## Student And Parent Experience

Students and parents should be able to:

- See an unofficial transcript from the dashboard.
- Filter by course status, academic period, and workspace when applicable.
- Download unofficial PDF if allowed by workspace policy.
- Request an official transcript or contact admin if self-service issue is disabled.
- See why an official transcript cannot be issued when a hold or missing data blocks it.
- Verify whether an older official transcript has been revoked/reissued.

Copy should be plain and careful. Use "unofficial transcript" for live records and "official transcript" only for issued snapshots.

## Reporting And Exports

Required exports:

- Official student PDF.
- Unofficial student PDF.
- Admin CSV for transcript lines.
- Admin CSV for issued transcript documents.

PDF requirements:

- Clear header and transcript type.
- Student identity and issue metadata.
- Course table with readable status and grade labels.
- Summary totals when configured.
- Verification block for official records.
- Footer with issuer, support contact, page number, and confidentiality note.

CSV exports should include stable IDs for reconciliation, but user-facing PDFs should avoid exposing internal database IDs unless needed.

## Search, Filters, And Admin Operations

Admins should be able to:

- Search by student name, account number, email, course, course key, status, and transcript document ID.
- Filter by workspace, academic period, status, hold state, issued date, and issuer.
- Bulk identify students with transcript warnings.
- Export a transcript audit report.
- Revoke or reissue official documents with required reason.

## API And Service Requirements

Future service methods should support:

- Resolve transcript preview for a student.
- List transcript warnings.
- Create official transcript draft.
- Issue official transcript.
- Download transcript file.
- Verify official transcript token.
- Revoke/reissue transcript.
- Manage transcript holds.
- Manage transcript policy.

Every service must enforce consumer, workspace, student, guardian, teacher, and admin permissions server-side.

## Acceptance Criteria

1. A student sees only their own unofficial transcript in the correct consumer/workspace.
2. A parent sees only linked children.
3. A teacher sees only assigned workspace students.
4. A workspace admin can preview and issue an official transcript for a workspace student.
5. Official transcript issue creates a locked snapshot and verification token.
6. Reissuing a transcript does not mutate the old official snapshot.
7. EduForTomorrow exports use EduForTomorrow branding and domains.
8. Institution exports use institution branding and verified institution domains.
9. Direct URL guessing cannot reveal another student's transcript or export.
10. Transcript warnings catch Moodle/local enrollment mismatches.
11. Grade and completion labels follow the workspace transcript policy.
12. Internal notes do not appear on official PDFs unless explicitly approved for transcript display.
13. Public verification confirms valid/revoked status without exposing full records.
14. Every issue, export, revoke, reissue, hold, and correction event is audited.

## Implementation Phases

The detailed build sequence is maintained in `docs/course-transcript-implementation-plan.md`.

Recommended high-level sequence:

1. Discovery, data mapping, and transcript risk register.
2. Transcript resolver using existing enrollment, Moodle enrollment, grade, completion, lesson progress, and live-session data.
3. Unofficial student, parent, teacher, and admin transcript views.
4. Workspace transcript policy settings.
5. Transcript warnings and data-quality readiness report.
6. Official snapshot tables, draft workflow, and issue action.
7. Official/unofficial PDF exports and admin CSV exports.
8. Holds, corrections, revoke, and reissue workflow.
9. Public verification.
10. APIs, scheduled maintenance, notifications, pilot rollout, and operational handoff.

## Test Matrix

Required tests:

- Student in EduForTomorrow cannot view Quran Academy transcript data.
- Institution parent cannot view another institution student's transcript.
- Teacher assigned to one workspace cannot view transcripts outside that workspace.
- Workspace admin can issue, revoke, and reissue an official transcript.
- Official snapshot remains unchanged after a course title, grade, or branding change.
- Verification URL reports valid for issued documents and revoked for revoked documents.
- Moodle enrollment without local offering appears as a warning, not a silent transcript line.
- Local approved request without Moodle enrollment appears as an approved-pending-sync warning.
- Course with hidden gradebook grade follows the workspace grade display policy.
- Official export excludes private notes and raw attempt data.
